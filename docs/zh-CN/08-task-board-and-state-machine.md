# 08. 任务看板与状态机

## 为什么需要任务看板

如果系统只有对话，没有任务状态，那么很多复杂请求会变得不可追踪。

尤其是以下场景：
- 需要多步执行
- 需要等待外部信息
- 需要多个专家协作
- 需要长期跟踪

因此 Town 应该维护一个统一任务看板。

## 核心状态

建议任务至少包含以下状态：
- `new`
- `routed`
- `executing`
- `waiting`
- `done`
- `archived`
- `failed`

## 状态含义

### new
任务刚创建，还没有分配专家。

### routed
Town 已经完成专家选择与上下文组装。

### executing
Hermes 正在执行。

### waiting
任务依赖外部输入，例如：
- 等待用户补充信息
- 等待外部 API
- 等待人工确认

### done
任务已经完成，并产出了可交付结果。

### archived
任务已归档，但仍可用于记忆和知识提取。

### failed
任务执行失败，需要重试、升级处理或人工介入。

## 为什么状态机重要

状态机不仅用于 UI 展示，还决定：
- 是否触发提醒
- 是否允许专家接管
- 是否应该学习入库
- 是否需要升级优先级

## 扩展建议

后续可以增加：
- `escalated`
- `reviewing`
- `scheduled`

但在 MVP 阶段，不要把状态设计得太复杂。

## 原则

任务状态必须：
- 可解释
- 可追踪
- 可恢复
- 能支持多端展示
