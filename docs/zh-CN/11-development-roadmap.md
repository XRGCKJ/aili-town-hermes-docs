# 11. 完整开发线路图

## 总体目标

Town × Hermes 的目标不是做一个单轮问答机器人，而是构建一个：
- 可持续运行
- 可多专家协作
- 可任务化追踪
- 可逐步学习沉淀
- 可多入口接入

的专家城镇系统。

因此开发应该按阶段推进，而不是一开始就追求全部能力同时完成。

## Phase 0：架构定稿

这一阶段的目标是把核心边界定义清楚。

### 需要明确的事情
- Town 与 Hermes 的职责边界
- Expert 的抽象模型
- Task 的生命周期
- Broadcast 的事件模型
- Memory / Knowledge 的分层
- 安全与治理边界

### 交付物
- 总体架构文档
- 核心模块说明
- 基础术语表
- 状态机定义
- 初版数据模型

## Phase 1：MVP 执行闭环

这一阶段的目标是跑通最小可用系统。

### 必要能力
- CLI 入口
- 用户提问 → Town 路由
- Town 组装上下文 → Hermes 执行
- Hermes 返回结构化结果
- Town 进行结果包装
- Task 状态流转可见

### 暂时不做
- 自动学习入库
- 复杂广播系统
- 多端 UI
- 大规模专家生态

### 验收标准
- 至少能稳定处理问答、调试、总结三类任务
- 能看到任务从 new 到 done 的状态变化
- 能按 expert profile 做基本路由

## Phase 2：专家协作与广播

这一阶段开始增强系统的“世界感”。

### 新增能力
- 多专家协作机制
- 广播事件流
- 专家订阅领域动态
- Town 公共公告板
- 简单任务升级机制

### 验收标准
- 外部事件可进入系统
- 不同专家可以接收不同广播
- 某些广播可触发新的任务

## Phase 3：学习闭环与知识沉淀

这一阶段开始让系统具备长期积累能力。

### 新增能力
- 候选知识提取
- 质量过滤
- 去重与归类
- 写入共享知识 / 专家知识
- 记忆摘要更新

### 验收标准
- 已完成任务可产生知识候选项
- 高价值信息可进入知识库
- 后续任务能消费这些知识

## Phase 4：多端交互与可视化

这一阶段增强产品可用性。

### 新增入口
- 飞书
- Web
- 后续可扩展 API

### 新增界面
- 专家大厅
- 任务看板
- 广播流
- 知识库视图
- 执行链路视图

## Phase 5：治理与运维增强

这是系统进入长期运行前必须完成的一步。

### 新增能力
- 工具权限矩阵
- 更细粒度预算控制
- 审计日志
- 回滚机制
- 失败恢复与人工介入

## 推荐策略

不要一开始就追求“全自动专家社会”。

正确顺序应该是：
1. 先跑通执行闭环
2. 再引入广播与协作
3. 再做学习沉淀
4. 最后做复杂多端与治理增强
